# RL 总览

这份文档说明本项目当前采用的 RL pipeline，包括角色分工、在线推理闭环、离线训练闭环以及模型部署方式。

## 系统目标

RL 部分的目标不是单纯让太阳能板一直追着最亮方向转，而是让系统在现场根据当前光照分布、当前姿态和当前能量预算，预测下一步应该把两个轴转到什么角度。  
在实现上，模型只负责给出下一步目标角度，底层执行、安全限制、能量预算维护和状态回传仍由 `STM32` 负责。

## 角色分工

### STM32

- 周期性采集现场状态
- 读取光照传感器，并维护当前两个步进电机角度的内部状态
- 读取两个 `INA219`，在本地计算 `panel_power`、`motor_power` 和 `energy_reserve`
- 执行树莓派下发的目标角度
- 回传执行状态、错误状态和训练所需附加日志
- 保留底层安全保护逻辑，例如角度限位、通信超时停止、动作幅度限制等

### Raspberry Pi

- 接收 `STM32` 上传的推理输入
- 执行在线 inference，即 `action = policy(state)`
- 输出下一步两个步进电机目标角度并下发给 `STM32`
- 保存现场日志，供后续离线训练使用
- 加载部署后的 `ONNX` 模型，在现场运行推理

### PC / 云端

- 不参与现场闭环控制
- 负责离线训练和策略更新
- 使用树莓派采集到的日志生成训练样本
- 训练完成后导出新模型，再部署回树莓派

## 在线推理闭环

当前正式 pipeline 的在线推理闭环如下：

1. `STM32` 周期性采集当前一圈光照传感器的光强、当前两个步进电机角度，以及当前归一化能量预算 `energy_reserve`。
2. `STM32` 将这些数据通过通信链路发送给树莓派。
3. 树莓派将数据整理成模型输入 `state`。
4. 模型执行一次前向推理，输出下一步两个步进电机的目标角度 `target_angle`。
5. 树莓派把目标角度发送回 `STM32`。
6. `STM32` 在本地完成角度执行、限位检查和状态回传。

这里要特别区分两层职责：

- 模型负责预测下一步目标角度
- `STM32` 负责把这个目标角度安全地执行出来

## inference 输入与输出

### 输入 state

模型 inference 时只使用以下信息：

- 一圈光照传感器的当前光强 `light_ring = [R1..Rn]`
- 当前两个步进电机角度 `current_angle = [theta1, theta2]`
- 当前归一化能量预算 `energy_reserve`

这里的 `current_angle` 定义为 `STM32` 已接受的上一步目标角度，也就是上一轮 inference 的输出在执行链路中的当前内部状态。  
这里的 `energy_reserve` 定义为 `STM32` 根据 `panel - motor - idle` 的功率时间积分估算出的归一化剩余能量预算。

### 输出 action

模型输出为：

- 下一步两个步进电机的目标角度 `target_angle = [target_theta1, target_theta2]`

当前版本中，`INA219` 原始电压、电流读数不进入 inference 输入，也不作为 reward 或训练日志字段。  
和能量相关的在线输入只有 `STM32` 计算好的单标量 `energy_reserve`，训练、reward 和评估只使用 `STM32` 已算好的 `panel_power`、`motor_power` 等派生量。

如果模型希望“不动”，则直接输出 `target_angle = current_angle`。

## 离线训练闭环

训练链路与现场控制链路是分开的。现场设备可以持续运行，训练则在 `PC` 或云端离线完成。

完整链路如下：

1. 树莓派在现场运行模型，同时保存交互日志。
2. 日志中除 `state` 和 `action` 外，还保留 reward 计算和评估所需的派生能量量与安全状态，但不保存 `INA219` 原始电压电流。
3. 这些日志被同步到 `PC` 或云端。
4. `PC` 或云端基于日志生成训练样本 `(state, action, reward, next_state, done)`。
5. 在 `PyTorch` 中完成离线训练并更新模型参数。
6. 训练完成后导出 `ONNX` 模型。
7. 将新的 `ONNX` 模型重新部署到树莓派，用于下一轮现场 inference。

当前训练策略采用两阶段方式，但部署接口保持不变：

- 第一阶段只训练模型学会追 `panel power`
- 第二阶段从第一阶段 checkpoint 继续微调，引入 `energy_reserve`、短时间窗净收益差和 hold 逻辑
- 无论训练处于哪个阶段，部署到树莓派的在线输入输出接口始终不变，都是 `light_ring + current_angle + energy_reserve -> target_angle`

## 两阶段训练方案

### 第一阶段

第一阶段的目标是先让模型学会追 `panel power`，也就是找到更高发电位置。

这一阶段的特点是：

- 模型输入已经使用最终部署接口
- reward 主体只关心 `panel_power`、`panel_power_delta` 和 `fault penalty`
- 暂时不让功耗和生存约束主导行为

训练完成后会保存一个 `stage1` checkpoint，人工评估其追光效果是否满意。

### 第二阶段

第二阶段从第一阶段 checkpoint 继续训练，不重新初始化。

这一阶段的目标是让模型学会三件事：

- 这一步是否值得移动
- 如果要移动，转到哪个角度更合理
- 在 `energy_reserve` 较低时，如何避免把系统能量预算耗尽

这一阶段的核心 reward 思路是：

- 比较“执行动作”和“保持不动”在短时间窗内谁的净收益更高
- 净收益按 `panel_energy - motor_energy - idle_energy` 计算
- `energy_reserve` 过低会触发生存风险和大惩罚
- 这些量优先由 `STM32` 在本地根据遥测先算好，再上传给树莓派用于日志与训练

## 模型部署路径

当前部署路径统一为：

`PyTorch training -> export ONNX -> ONNX Runtime on Raspberry Pi`

这样做的原因是：

- 训练与运行解耦，树莓派不承担训练负担
- 树莓派现场只做前向推理，运行负担更稳定
- 模型版本可以独立迭代，不影响 STM32 的底层实时控制逻辑

## 为什么这样分层

当前架构的核心考虑有四点：

- `STM32` 的实时性和安全性不能被训练逻辑干扰
- 树莓派适合承担模型加载、前向推理和日志整理
- 离线训练放在 `PC/云端`，便于迭代模型和比较不同策略
- `STM32` 更适合承担 `energy_reserve` 这类依赖底层遥测和控制周期的内部状态维护

因此，本项目把现场控制链和训练链明确分开，但又通过日志与模型部署把两者连接起来。
